实现细节

function compose(...funcs) {
  if (funcs.length === 0) {
    return arg => arg
  }

  if (funcs.length === 1) {
    return funcs[0]
  }

  return funcs.reduce((a, b) => (...args) => a(b(...args)))
}

在applyMiidleWare()中middleware({ dispatch, getState })将store => next => action => {}形式的中间件通过闭包获得对dispatch, getState的访问，再由compose得到a(b(c(d(store.dispatch))))最终的函数，这种reduce的使用方式值得学习。

写写中间件

利用中间件来解决reducer数据依赖复杂的问题是在墨刀刚学到的。

开始的我们

以前刚开始接触redux的时候，为了图方便，会在局部的reducer里读取store上整体的state，如以下代码

const statusReducer = (state = initialState, { type, payload }) => {
  switch (type) {
    case 'set:status:count':
      return {
        ...state.status,
        count: state.count + payload.count
      }
    default:
      return state.status
  }
}

const countReducer = () => ({}) // some easy reducer

const reducer = (state, action) => ({
  status: statusReducer(state, action),
  count: countReducer(state.count, action)
})

这样的reducer works well，但是为了读取到state，reducer的逻辑比较复杂，维护起来比较脏，我们可以写一个中间件，来处理对state的数据依赖。

来一个中间件

const captureFunctionMiddleWare = ({ getState, dispatch }) => next => action => {
  if (typeof action === 'function') {
    return action(getState, dispatch, next)
  }
  next(action)
}

const setCount = count => ({ getState, dispatch }) => {
  const {
    count: stateCount
  } = getState()
  dispatch({
    type: 'set:status',
    payload: {
      count: count + stateCount
    }
  })
}

const statusReducer = (state = initialState, { type, payload }) => {
  switch (type) {
    case 'set:status:count':
      return {
        ...state,
        count: payload.count
      }
    default:
      return state
  }
}

const reducer = combineReducers({
  status: statusReducer,
  count: countReducer
})

我们使用captureFunctionMiddleware来对函数action进行捕获，给其传递(getState, dispatch, next)，这样写利用中间件获取到整体的state，不需要在reducer中进行hack，reducer会干净很多。

更进一步

使用上述的中间件确实解决了reducer很脏的问题，但是问题来了，action creator现在变得比较复杂了，能不能把逻辑拆出来呢，我们再写一个trap middleware creator。
